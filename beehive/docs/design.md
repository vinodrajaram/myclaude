## Design Specification

### Visual Language

**Aesthetic:** Warm professional. Not cold fintech. Not consumer-bright. The "Beehive"
name earns a warm accent — amber/gold — used sparingly against an otherwise minimal,
high-information-density interface. Think Linear meets a banker's Bloomberg terminal:
dense, fast, trustworthy, with one warm note that says "this was made with care."

**Typography:**
- Display / headings: Söhne (or Inter as fallback) — geometric, professional, legible at density
- Body / data: Inter — neutral, extremely readable at small sizes
- Monospace (timestamps, IDs): JetBrains Mono
- Base size: 13px (bankers work in dense UIs; don't make it feel like a consumer app)

**Color system (dark default, light toggle):**

Dark (default):
- Background: `#0F0F0F`
- Surface: `#1A1A1A`
- Surface elevated: `#242424`
- Border: `#2E2E2E`
- Text primary: `#F0EEE8` (warm white)
- Text secondary: `#8A8A8A`
- Accent: `#C9932A` (amber — the hive)
- Destructive: `#E05252`
- Success: `#52A87A`

Light (toggle):
- Background: `#F7F5F1` (warm off-white, not stark white)
- Surface: `#FFFFFF`
- Surface elevated: `#FAFAF8`
- Border: `#E4E2DC`
- Text primary: `#111110`
- Text secondary: `#6B6966`
- Accent: `#A67C1E` (amber, darkened for contrast on light)
- Destructive: `#C0392B`
- Success: `#2E7D52`

Theme persisted to `localStorage`. FOUC prevention: inline script on `<html>` applies theme class before first paint.

**Spacing scale:** 4px base unit. 4/8/12/16/24/32/48px. No half-pixels.

**Border radius:** 6px for cards, 4px for inputs, 2px for badges. Not rounded-full — professional, not playful.

**Shadows:** Subtle. `0 1px 3px rgba(0,0,0,0.4)`. Elevation via border, not shadow.

## Problem Statement

Investment and corporate banking VPs/Directors go back-to-back through 8+ calls per
day — Zoom video, Zoom phone, mobile, and in-person — with no structured way to
capture what was discussed. The result: missed action items, forgotten follow-ups, and
relationship damage when a committed deliverable slips. There is no standard system.
Every banker has a different broken workaround (handwritten notes that never get
transcribed, quick self-emails, or nothing at all). A missed follow-up to a CFO isn't
an embarrassment — it's a mandate at risk.

## Demand Evidence

A banker described the pain directly and unprompted: going back-to-back between
meetings, not taking notes, no AI summary, and missing follow-ups as a result. This is
not a leadership mandate or a logical extrapolation — it is firsthand behavioral
evidence of a real workflow gap. The banker would be upset if this problem disappeared
because it costs them: professionally (missed follow-ups damage client trust) and
operationally (no record of what was agreed on a deal).

## Status Quo

Bankers currently use a mix of three broken approaches:
1. **Nothing** — rely on memory until it fades
2. **Handwritten notes** — on legal pads, rarely transcribed, often lost
3. **Quick self-emails** — "Reminder: send model to James" — incomplete and too late

No standard exists. Each banker operates differently. This means there is no incumbent
habit to displace — the product is not asking bankers to change what they do, it is
giving them something they have never had.

## Target User & Narrowest Wedge

**Primary user:** VP or Director at a large bank. Manages multiple active deals,
owns client relationships. Has 6–10 calls per day across call types. A missed
follow-up damages a relationship built over years. Gets promoted by execution
quality and client trust; gets fired (or loses deals) when commitments slip.

**Narrowest wedge:** Auto-capture Zoom calls with zero effort.
- Banker connects Zoom OAuth once; app watches for call end
- When call ends: app fetches cloud recording via Zoom API, submits to Google Cloud Speech-to-Text
- 30–90 seconds later: structured summary (3–5 sentences) + LLM-extracted action items
- Action items: LLM extracts commitments and owners from transcript; banker can
  add/edit/delete in-app; app is the primary store in V1 (no CRM/tasks sync yet)
- Primary CTA: one-tap "Draft Follow-up Email" — pre-written editable draft, recipients
  pre-filled from Zoom attendee list; banker reviews and sends via Outlook (Windows:
  MAPI/COM; Mac: mailto deep-link; fallback: copy to clipboard)

This is the highest-volume, most structured call type. In-person and mobile calls
come in V2 once trust is established.

## Constraints

1. **Compliance / data residency — PREREQUISITE, NOT PARALLEL:**
   Banking calls contain MNPI, client names, and deal terms — all regulated. Audio and
   transcripts must not touch public LLM APIs. MVP build does NOT start until preliminary
   compliance sign-off is obtained on a reference architecture. **Confirmed LLM stack:
   Google Gemini via Vertex AI** (e.g., Gemini 1.5 Pro or Gemini 2.0 Flash), with
   **Google Cloud Speech-to-Text** for transcription — both deployed within the bank's
   Google Cloud organization with VPC Service Controls and data residency enforced to a
   single region (e.g., us-central1 or eu-west). This is a unified Google Cloud stack:
   one vendor, one data boundary, one compliance conversation. On-device Whisper.cpp +
   local Llama is the fallback if GCP approval is delayed. Required sign-offs: data
   residency region, retention policy (how long audio/transcripts stored), encryption
   in transit and at rest, audit log requirements (Cloud Audit Logs covers this
   automatically on GCP), and MNPI handling policy. Estimated compliance timeline:
   2–4 weeks to get preliminary answer.

2. **Trust is the product:** Bankers are skeptical of recording tools. If the first
   output is wrong or feels insecure, they won't return. "Privacy-first" is not a
   feature — it is the architecture. The UI must communicate clearly at every step:
   what was recorded, where it went, and that nothing left without explicit consent.
   On the recording screen, display: "Audio is processed via [Bank Name] Google Cloud environment.
   Nothing is sent to external services."

3. **Zero effort is the bar:** Bankers will not adopt a tool that requires setup per
   call. Calendar integration (Outlook/Exchange) is optional convenience — recording
   works without it. When connected: pre-fill call title and attendees from the next
   calendar event. When not connected: banker types call name manually (< 10 seconds).
   Calendar is never required to start recording.

4. **Zoom integration path — async recording API (not real-time SDK tap):**
   Real-time SDK audio capture from inside Zoom requires either a Zoom plugin
   (admin approval needed) or system-level audio capture (fragile, Zoom-version-dependent).
   V1 uses **Zoom's Meeting Recording API** — after the call ends, the app fetches the
   cloud recording (MP4/M4A), submits it to Google Cloud Speech-to-Text for transcription, and
   delivers the summary. This requires the banker to have Zoom cloud recording enabled
   (standard for licensed Zoom accounts at most banks) and a one-time Zoom OAuth
   connection. No admin approval needed for the OAuth path; admin approval only needed
   if the bank wants to auto-enable recording for all users.

5. **Latency expectations (revised):** Given async recording fetch, summary will be
   delivered 30–90 seconds after call ends, not 15 seconds. Progress states shown to
   banker: "Fetching recording → Transcribing → Summarizing → Done." This is still
   dramatically better than the status quo (no notes at all). 15-second target is
   achievable only with on-device Whisper in V2; not a V1 constraint.

6. **Multi-channel scope (V1 → V2):**
   - V1: Zoom video via Zoom Recording API (async, OAuth)
   - V2: Zoom phone (dial-in recording), mobile calls (system audio), in-person
     (microphone + post-meeting dictation)
   Each channel has different capture mechanics; V1 should not over-engineer for V2.

7. **Transcript visibility:** Transcription happens post-call (not real-time in V1).
   The full transcript is stored but hidden by default — bankers see only the 3–5
   sentence summary and action items. "View full transcript" is available as an
   expandable section. Bankers will not review raw transcripts; they want output.

## Premises

1. The core pain is back-to-back meetings with no capture. The VP walks out of 8
   calls with nothing written down. Missed follow-ups cost real deals.
2. The wedge is Zoom auto-capture first. It covers the highest-volume, most
   structured call type. In-person and mobile come later.
3. Privacy-first is the core differentiator — not a compliance checkbox. "Nothing
   leaves this device without your explicit consent" is what gets bankers to use it
   on real client calls in week one.
4. The LLM must run on approved internal infrastructure. No client call data goes
   to a public API. Non-negotiable.
5. The output that matters is NOT the transcript. It's the follow-up email draft and
   action item list. Bankers don't want to read a 40-minute transcript — they want
   to know what they promised and to whom.

## Approaches Considered

### Approach A: Zero-friction Zoom Bot
Summary: A Zoom bot (like Otter or Fireflies) joins calls automatically, transcribes,
and delivers a structured summary + action items via email/Teams when the call ends.
Hosted on internal infra.
Effort: M — human: ~3 months / CC: ~2 weeks
Risk: Med — Zoom bot admin approval, client distrust of bots joining calls
Pros: Truly zero effort; covers every Zoom call automatically; familiar UX pattern
Cons: Requires Zoom admin approval; clients may object to a bot; no in-person/mobile
coverage

### Approach B: Privacy-First Local App (CHOSEN)
Summary: Desktop + mobile app the banker controls. Tap Record before a call. Audio
stays on-device or routes to internal bank servers. AI processes via internal API.
Summary + action items delivered on demand within 15 seconds of stopping.
Effort: L — human: 4–5 months / CC: ~3 weeks
Risk: Med — on-device vs. server-side quality tradeoff; multi-platform
Pros: Maximum privacy; banker controls every recording; covers ALL call types;
"nothing left this device without your permission" is a concrete, verifiable claim
Cons: Requires deliberate Record tap (small friction vs. auto-bot); more complex
cross-platform

### Approach C: Copilot Layer on Existing Tools
Summary: Browser extension + mobile app that sits on top of Zoom, Teams, and phone
dialer. Intercepts audio with consent, routes to internal transcription/LLM API,
surfaces summary inline.
Effort: L-XL — human: 5–6 months / CC: ~4 weeks
Risk: High — extension approval inside the bank is slow; audio API access is
inconsistent across OS/browser
Pros: No new app to open; covers Zoom + Teams + phone in one product
Cons: Most complex; extension approval processes slow; audio capture unreliable

## Technical Implementation Notes

### Call End Detection
Zoom Recording API does not emit webhooks when a recording becomes available. The app
polls `GET /meetings/{meetingId}/recordings` at 15-second intervals for up to 5 minutes
after the banker taps "End Call" (or after the Zoom client meeting window closes, detected
via local process monitoring). On recording availability: fetch MP4/M4A, submit to Google Cloud Speech-to-Text
Whisper. If no recording appears after 5 minutes: surface error UI — "Recording not found.
Was cloud recording enabled?" — with a link to enable it. Retry manually available.

### Recipient Pre-fill for Follow-up Email
After call ends, fetch `GET /meetings/{meetingId}/participants` from Zoom API (returns
names and emails of attendees who joined via Zoom). Pre-fill email recipients from this
list, excluding the banker themselves. If a participant joined by phone (no email), show
their number and a name field for the banker to fill. If no email is available for any
participant, the "Draft Follow-up Email" CTA opens with recipients blank and a tooltip:
"Add recipients — not all attendees had Zoom email addresses."

### Transcript Storage & Retention
Transcripts stored in encrypted local SQLite database on the banker's device
(AES-256, key derived from OS keychain). Audio files are NOT stored locally — only the
transcript text and summary. Audio is submitted to Google Cloud Speech-to-Text over TLS and deleted
from GCS after transcription (Google Cloud Speech-to-Text does not retain audio by default; confirm
with Google Cloud documentation). Transcript retention: indefinite on-device (banker controls
deletion). Compliance may require a 90-day auto-delete policy; this is a field in app
settings, defaulting to "Never" until compliance specifies otherwise.

### Audio Transmission Consent UI
Before any audio is sent to Google Cloud Speech-to-Text, the app surfaces a one-time consent dialog
on first use: "This call's recording will be sent to [Bank Name]'s Google Cloud environment for
transcription. Audio is not stored after transcription." After first consent, subsequent
calls show a non-blocking status bar: "Transcribing via [Bank Name] Google Cloud · Tap for
details." Banker can revoke at any time in Settings. Per-call opt-out: if banker stops
recording and taps "Delete" before the summary is requested, no audio is transmitted.

### Action Item Extraction
LLM prompt instructs Gemini 1.5 Pro (Vertex AI) to extract commitments expressed as future obligations.
Extraction rules: (1) extract only explicit commitments ("I will", "we'll send",
"can you", "I'll follow up"); (2) assign owner based on who made the commitment
(speaker attribution from Whisper diarization, if available); (3) if ownership is
ambiguous, default owner to "You" (the banker) and flag as uncertain. Banker can
edit/delete any extracted item in the action item list before saving. False positives
are preferable to false negatives — extracting too many is better than missing one.
No automated feedback loop in V1; quality improvement via prompt iteration.

### OAuth Token Lifecycle
Zoom OAuth access token expires every 1 hour; refresh token valid 90 days.
App uses refresh token to silently renew before expiry. If refresh token expires
(banker hasn't used the app in 90+ days): re-prompt for Zoom login. Token stored in
OS keychain (Keychain on Mac, Windows Credential Manager on Windows). Token is scoped
to `recording:read` and `meeting:read` only — no write permissions requested.

### Compliance Contingency
If compliance sign-off on GCP path is delayed beyond 4 weeks: pivot to on-device
processing as a temporary path. On-device stack: Whisper.cpp (medium model, ~500MB,
English-only) for transcription + Llama 3.1 8B (Q4 quantized, ~5GB) for summarization.
This path produces lower quality transcripts (especially for finance vocabulary) but
enables a working demo for the VP while compliance resolves. On-device path can be
built in parallel with the GCP integration at ~20% additional effort.

## Knowledge System (Beehive Intelligence Layer)

Every call captured by Beehive is not just a note — it's a node in the banker's
personal deal memory. Over time, the hive builds a connected intelligence: who said
what, what was promised, how deals evolved across calls.

### Architecture

After each call is summarized, a background pipeline runs:

1. **Entity extraction** — Gemini extracts structured entities from the summary:
   - People: "James Okafor", "Sarah Mitchell"
   - Companies: "Riverview Capital"
   - Deals: "Q2 Financing", "Refi Discussion"
   - Topics: "leverage ratio", "go-to-market timeline"
   Stored in a new `entities` table (call_id, type, value).

2. **Embedding** — The summary + action items are embedded using
   `text-embedding-004` (Gemini embedding model). The resulting `vector(768)`
   is stored in a new `embedding` column on `transcripts`.

3. **pgvector index** — A `ivfflat` index on the embedding column enables
   sub-second semantic similarity search across all of a banker's calls.
   No new service: pgvector is a PostgreSQL extension, enabled on Cloud SQL.

### What this enables

- **Context injection at summary time:** When a new call is processed, Beehive
  retrieves the top-3 most semantically similar past calls and injects them into
  the Gemini summarization prompt. The summary says: "This is your 4th call with
  Riverview Capital. Previously: [prior context]." Continuity, not isolation.

- **Semantic search:** Banker types "leverage ratio" → Beehive embeds the query
  → cosine similarity search → returns ranked calls where this topic arose, even
  if the exact words differ.

- **Natural language Q&A:** Banker asks "What did I commit to with Meridian this
  month?" → embed query → retrieve top-N relevant call chunks → Gemini answers
  with citations to specific calls.

- **Client/deal thread:** All calls tagged with the same company or deal are
  automatically grouped into a timeline view. The banker sees the full arc of a
  relationship, not individual isolated calls.

### Database additions

```sql
-- Enable pgvector (one migration, no new service)
CREATE EXTENSION IF NOT EXISTS vector;

-- Add embedding to transcripts
ALTER TABLE transcripts ADD COLUMN embedding vector(768);
CREATE INDEX transcripts_embedding_idx ON transcripts
  USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);

-- Entity extraction store
CREATE TABLE entities (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  call_id     UUID REFERENCES calls(id) ON DELETE CASCADE,
  type        TEXT NOT NULL,  -- person | company | deal | topic
  value       TEXT NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT now()
);
CREATE INDEX entities_call_id_idx ON entities(call_id);
CREATE INDEX entities_type_value_idx ON entities(type, value);
```

### Name rationale

Every bee returns to the hive with information. The hive — not any individual bee —
is where intelligence lives. Each call is a bee. Beehive is the connected memory
that makes the whole more than the sum of its parts.
